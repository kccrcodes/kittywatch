# Specs

Feature work for the remaining Milestone 1/2 backend issues
(`docs/INITIAL_ISSUES_MILESTONES.md`) gets a spec folder here as it's built,
each with:

- `requirements.md` — user story + acceptance criteria
- `design.md` — approach, API shape, data flow
- `tasks.md` — checklist of implementation steps

Example: `.kiro/specs/sighting-submission/`. Not every issue needs a full
spec — reserve this for the two hero features (re-ID, disappearance alert)
and anything with a non-obvious design decision worth writing down before
coding. Simple CRUD routes can skip straight to implementation.
