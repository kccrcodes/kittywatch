**Yes — significantly.** `SPEC.md`, `ROADMAP.md`, and `design.md` are newer and are the actual source of truth the team is building against; they supersede several core decisions in the architecture doc. The repo's git history confirms the team is already executing the newer plan, not the original one.

| # | Area | `KITTYWATCH_ARCHITECTURE.md` said | Current source of truth (`SPEC.md`/`ROADMAP.md`/`design.md`) says | Why it matters |

## 3. Issues

Each issue is scoped to be a single GitHub issue. Labels: `frontend`, `backend`, `ai`, `security`, `docs`, `infra`. Priority tags follow ROADMAP's `[MUST]` / `[SHOULD]` / `[CUT]` convention.

### Milestone 0 — Day-one contract & de-risking

- [ ] **Freeze schema + API contract** — `docs` `infra` · `[MUST]` · Both
  Confirm `SPEC.md`'s data model and `GET/POST /api/cats`, `POST /api/sightings` shapes are final. No changes without notifying the other side.

- [ ] **De-risk CLIP re-identification** — `ai` · `[MUST]` · Backend/AI
  Throwaway script against the HuggingFace Inference API (`openai/clip-vit-base-patch32`): score same-cat photo pairs vs. different-cat pairs. If it doesn't discriminate on the free tier, pivot to Replicate or local CLIP **today** — every downstream feature depends on this working.

- [ ] **Supabase project setup** — `backend` `infra` · `[MUST]` · Backend/AI
  Create `users`, `cats`, `sightings`, `alerts` tables per `SPEC.md`; enable `pgvector`; create the Storage bucket for photos.

- [ ] **`match_cat_embedding` RPC** — `backend` `ai` · `[MUST]` · Backend/AI
  Postgres function: cosine similarity `1 - (embedding <=> query_embedding)` against a cat's stored embeddings, best match only.

### Milestone 1 — Core loop

- [ ] **`POST /api/cats` — register a cat** — `backend` `ai` · `[MUST]` · Backend/AI
  Fuzz coordinates (±0.0005°/~50m) → insert cat → generate founding CLIP embedding → store founding sighting (`match_score: 1.0`) per `SPEC.md`.

- [ ] **`GET /api/cats` (bounding box) + `GET /api/cats/[id]`** — `backend` · `[MUST]` · Backend/AI
  Bounding-box query by `lat`/`lng`/`radius` for map pins; full profile with joined sighting history for the profile route.

- [ ] **`POST /api/sightings` — submit a sighting** — `backend` `ai` · `[MUST]` · Backend/AI
  Rate-limit → generate embedding → pgvector match → insert sighting → update cat's `last_seen_at`/`status`. **Stub `match_score`** if CLIP isn't wired yet so frontend isn't blocked.

- [ ] **Map screen: Leaflet + colour pins + geolocation** — `frontend` · `[MUST]` · Frontend/Design
  Full-bleed map, colour-coded by `STATUS`, browser Geolocation API centring, FAB "Register" in thumb reach.

- [ ] **Cat profile bottom sheet** — `frontend` · `[MUST]` · Frontend/Design
  Slides up on pin tap: photo gallery, sighting history, status badge. Wire to `GET /api/cats/[id]` in place of `lib/mock-data.ts`.

- [ ] **Camera submission flow** — `frontend` · `[MUST]` · Frontend/Design
  `<input type="file" accept="image/*" capture="environment" />`, upload to Supabase Storage first, then POST the resulting `photo_url` to `/api/sightings`.

- [ ] **Cat registration flow** — `frontend` · `[MUST]` · Frontend/Design
  Drop pin + form + founding photo, wired to `POST /api/cats`. `register-cat-card.tsx` exists — confirm it's the registration entry point or needs adapting.

- [ ] **Hardcode demo user/device id** — `backend` · `[MUST]` · Both
  Don't let auth block the core loop; real OAuth is out of scope for this hackathon window.

### Milestone 2 — The two hooks, demoable

- [ ] **Surface `match_score` from the re-ID RPC in the sighting response** — `backend` `ai` · `[MUST]` · Backend/AI

- [ ] **Re-ID reveal animation** — `frontend` · `[MUST]` · Frontend/Design
  Ring filling to "92% match" on submit — the single most demo-able moment per `design.md` §7. Design deliberately; this is the whole technical story on video.

- [ ] **Disappearance detection: cron + manual trigger** — `backend` · `[MUST]` · Backend/AI
  Supabase Edge Function, daily, flips cats past `MISSING_THRESHOLD` (7 days) to `missing`, writes an `alerts` row. **Also expose `POST /api/admin/run-check`** with a configurable threshold — you cannot wait 7 days in the demo video. *(Note: this replaces the architecture doc's perpetual Temporal workflow — see §1.)*

- [ ] **Missing-cat state (pin + profile)** — `frontend` · `[MUST]` · Frontend/Design
  Must land emotionally (Theme + UX), not just informationally — this is where the pitch's "digital heartbeat" narrative is felt or lost.

- [ ] **Rate limiting** — `backend` `security` · `[MUST]` · Backend/AI
  Upstash Redis, 10 submissions/user/hour, enforced before any write endpoint runs its logic.

- [ ] **Optimistic update on sighting submit** — `frontend` · `[SHOULD]` · Frontend/Design

- [ ] **Web Push notifications (or fallback)** — `backend` · `[SHOULD]` · Backend/AI
  Try Web Push; fall back fast to Resend email or an in-app banner if VAPID/iOS setup eats time.

- [ ] **Temporal — stretch goal only** — `backend` `infra` · `[CUT]` (do only if ahead) · Backend/AI
  If M1/M2 MUSTs are done early, wrap the disappearance check in a Temporal Cloud workflow for the sponsor-bonus points. Do not start this before the core loop and both hero moments are working — see §1, item 1.
